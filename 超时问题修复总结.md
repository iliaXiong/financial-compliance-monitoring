# 超时问题修复总结

## 🎯 问题诊断

通过SQL诊断发现核心问题：

**错误信息**: `Error: Execution Timeout - task did not complete in time`

### 问题详情
- ✅ 任务执行开始了
- ✅ 网站访问成功了

- 原因：LLM API响应太慢

```

创建时间: 2026-04-29 02:39:48
```

## 🔧 修复方案

### 修复1: 增加任务执行超时时间
**文件**: `backend/src/services/TaskScheduler.pgboss.ts`

**修改**:
```typescript
// 从 2分钟 增加到 5分钟
const orchestrationResult = await this.subagentOrchestrator.executeParallel(
  task.targetWebsites,
  task.keywords,
  300000 // 5 minutes timeout (increased from 2 minutes)
);
```

**原因**: 给LLM API更多时间来处理请求和返回结果

### 修复2: 增加LLM API调用超时时间
**文件**: `backend/src/services/ContentRetriever.ts`

**修改**:
```typescript
// 从 60秒 增加到 120秒
const response = await axios.post(
  this.llmApiUrl,
  requestBody,
  {
    headers,
    timeout: 120000 // 120 seconds for LLM (increased)
  }
);
```

**原因**: Webull的Claude API可能响应较慢，需要更长的超时时间

### 修复3: 添加详细的时间日志
**文件**: `backend/src/services/ContentRetriever.ts`

**添加**:
```typescript
const startTime = Date.now();
console.log(`[ContentRetriever] Calling LLM API...`);
const llmResponse = await this.callLLM(prompt);
const llmDuration = Date.now() - startTime;
console.log(`[ContentRetriever] LLM API responded in ${llmDuration}ms`);
```

**原因**: 可以在Railway日志中看到LLM API的实际响应时间，便于后续优化

```sql
  website_url,
  keyword,
  found,
  COALESCE(context, '(无错误信息)') as 错误或上下文,
  created_at
FROM retrieval_results 
WHERE execution_id = (
  SELECT id 
  FROM executions 
  ORDER BY start_time DESC 
  LIMIT 1
)
ORDER BY created_at;
```

## 🔍 后续优化建议

### 优化1: 缓存网站内容
如果同一个网站被多次访问，可以缓存内容，避免重复获取。

### 优化2: 优化LLM提示词
减少发送给LLM的内容量，可以加快响应速度。

### 优化3: 使用更快的LLM模型
如果Webull的Claude响应太慢，可以考虑：
- 使用更快的模型（如GPT-3.5）
- 或者配置备用LLM API

### 优化4: 并行处理优化
当前是串行处理：获取内容 → LLM搜索 → 保存结果
可以考虑部分并行化。
