# 任务执行流程图

## 完整执行流程

```
用户创建任务
    ↓
TaskScheduler.executeTask()
    ↓
创建 Execution 记录 (status: running)
    ↓
SubagentOrchestrator.executeParallel()
    ↓
    ├─→ ContentRetriever.retrieveFromWebsite(website1, keywords)
    │       ↓
    │   WebsiteAnalyzer.analyze() [如果启用]
    │       ↓
    │   获取页面链接和文档链接
    │       ↓
    │   fetchPageContent(mainPage) - 使用Jina Reader
    │       ↓
    │   fetchSubPages(pageLinks) - 使用Jina Reader
    │       ↓
    │   fetchDocuments(documentLinks) - 使用Jina Reader
    │       ↓
    │   llmWebSearch() - 调用LLM搜索关键词
    │       ↓
    │       ├─→ buildLLMSearchPrompt() - 构建提示词
    │       ├─→ callLLM() - 调用LLM API
    │       └─→ parseLLMSearchResponse() - 解析LLM响应
    │               ↓
    │           返回 KeywordMatch[] 和 DocumentResult[]
    │
    ├─→ ContentRetriever.retrieveFromWebsite(website2, keywords)
    │   ...
    │
    └─→ 返回所有网站的检索结果
        ↓
保存检索结果到数据库 (retrieval_results表)
    ↓
    ├─→ 每个网站 × 每个关键词 = 1条记录
    │   - found: true/false
    │   - content: 如果found=true
    │   - context: 错误信息或上下文
    │   - source_url: 来源URL
    │   - document_url: 文档URL（如果有）
    │
    └─→ 如果网站访问失败，为每个关键词创建failed记录
        ↓
AnalysisService.generateSummary()
    ↓
    └─→ 保存到 summary_documents 表
        ↓
AnalysisService.compareResults()
    ↓
    └─→ 保存到 comparison_reports 表
        ↓
AnalysisService.analyzeCrossSite()
    ↓
    └─→ 保存到 cross_site_analyses 表
        ↓
更新 Execution 记录 (status: completed)
    ↓
完成
```

## 关键检查点

### 检查点1: Execution记录
**位置**: `executions` 表
**检查**: 
- status 是否为 'completed'
- error_message 是否为 null
- 执行时长是否合理（30-120秒）

### 检查点2: 检索结果
**位置**: `retrieval_results` 表
**检查**:
- 是否有记录（total_results > 0）
- found 字段的值
- 如果 found=false，查看 context 字段的错误信息

### 检查点3: LLM调用
**位置**: Railway日志
**检查**:
- `[ContentRetriever] Using LLM to search for keywords`
- `[ContentRetriever] LLM response (first 500 chars)`
- `[ContentRetriever] LLM search completed`
- `[ContentRetriever] Keyword matches`

### 检查点4: 分析结果
**位置**: `summary_documents`, `comparison_reports`, `cross_site_analyses` 表
**检查**:
- 是否生成了摘要文档
- 是否生成了对比报告（第二次执行后）
- 是否生成了跨站点分析

## 常见失败点

### 失败点A: 网站访问失败
**症状**: 
- retrieval_results 中 found=false
- context 显示 "Error: Connection timeout" 或 "Error: HTTP 403"

**原因**:
- 网站无法访问
- Jina Reader API失败
- 网站阻止爬虫

**解决**:
- 检查网站是否可访问
- 尝试直接访问 https://r.jina.ai/[网站URL]
- 查看Railway日志中的详细错误

### 失败点B: LLM调用失败
**症状**:
- Railway日志显示 "LLM search failed"
- 或者没有 "LLM response" 日志

**原因**:
- LLM API配置错误
- LLM API密钥无效
- LLM API URL错误
- 网络连接问题

**解决**:
- 检查Railway环境变量
- 测试LLM API是否可访问
- 查看详细错误日志

### 失败点C: LLM响应解析失败
**症状**:
- Railway日志显示 "Failed to parse LLM response"
- 回退到简单关键词匹配

**原因**:
- LLM返回的不是有效JSON
- LLM返回格式不符合预期

**解决**:
- 查看LLM响应内容
- 检查LLM模型是否支持JSON输出
- 可能需要调整提示词

### 失败点D: 关键词确实不存在
**症状**:
- retrieval_results 中 found=false
- context 为空或显示 "No matches found"
- LLM正常调用，但返回 found=false

**原因**:
- 关键词在网站中确实不存在
- LLM理解错误

**解决**:
- 手动检查网站是否包含关键词
- 尝试不同的关键词
- 检查关键词拼写

## 诊断优先级

1. **首先检查**: Execution记录的status和error_message
2. **然后检查**: retrieval_results的数量和found字段
3. **接着检查**: Railway日志中的LLM相关日志
4. **最后检查**: 分析结果是否生成

按照这个顺序，可以快速定位问题所在。
