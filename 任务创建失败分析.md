# 任务创建失败分析

## 问题现状

### 数据库状态
- ✅ 演示用户存在（ID: `00000000-0000-0000-0000-000000000000`）
- ✅ 数据库表结构完整（tasks, executions, retrieval_results等）
- ❌ **没有任何任务记录**
- ❌ **没有任何执行记录**
- ❌ **没有任何检索结果**

### 问题判断
用户在前端创建了任务，但任务没有被保存到数据库。这说明：
1. 前端请求可能未到达后端
2. 后端处理请求时出错
3. 数据库写入失败
4. TaskScheduler初始化失败导致整个创建流程中断

## 根本原因分析

### 最可能的原因：TaskScheduler未正确启动

我们之前修复了TaskScheduler的初始化问题：
- 修改了 `backend/src/routes/tasks.ts` 使用依赖注入
- 修改了 `backend/src/index.pgboss.ts` 注入TaskScheduler实例

但是，**Railway可能还没有重新部署最新的代码**。

### 其他可能的原因

#### 1. pg-boss schema未创建
**症状**: TaskScheduler.start() 失败

**原因**: 
- pg-boss需要在数据库中创建 `pgboss` schema
- 如果schema不存在，pg-boss无法启动
- TaskScheduler启动失败会导致整个服务启动失败

**验证方法**:
执行 `检查pgboss表.sql` 查看pgboss schema是否存在

#### 2. 数据库连接权限问题
**症状**: 无法创建pgboss schema

**原因**:
- Supabase用户可能没有创建schema的权限
- Connection Pooler可能有限制

**解决方案**:
手动创建pgboss schema或使用直连端口

#### 3. Railway环境变量未更新
**症状**: 后端使用旧的配置

**原因**:
- `DEMO_MODE` 未设置
- 其他环境变量缺失

**验证方法**:
检查Railway环境变量配置

#### 4. 代码未重新部署
**症状**: 后端仍在运行旧代码

**原因**:
- GitHub推送后Railway未自动部署
- 部署失败但未注意到

**验证方法**:
检查Railway部署历史和日志

## 诊断步骤

### 步骤1: 检查Railway部署状态

1. 访问 Railway Dashboard
2. 选择后端服务
3. 查看 "Deployments" 标签
4. 确认最新的提交是否已部署：
   - 最新提交: `20a7672` "Fix root vercel.json..."
   - 或更早的: `bd3d0a0` "Fix TaskScheduler initialization..."

### 步骤2: 检查Railway部署日志

1. 点击最新的部署
2. 查看 "Build Logs"
3. 查看 "Deploy Logs"
4. 搜索关键信息：
   - `✅ Task scheduler started (pg-boss)`
   - `TaskScheduler not started`
   - 任何错误信息

### 步骤3: 检查pg-boss表

在Supabase SQL Editor中执行 `检查pgboss表.sql`：
- 查看pgboss schema是否存在
- 查看pgboss表是否创建
- 查看是否有job记录

### 步骤4: 测试任务创建API

使用curl或Postman测试API：

```bash
curl -X POST https://financial-compliance-monitoring-production.up.railway.app/api/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "name": "测试任务",
    "keywords": ["测试"],
    "targetWebsites": ["https://example.com"],
    "schedule": {
      "type": "once",
      "time": "10:00"
    }
  }'
```

查看响应：
- 200: 成功，返回任务ID
- 500: 服务器错误，查看错误信息
- 其他: 根据错误码判断问题

## 解决方案

### 方案1: 确保Railway已部署最新代码

如果Railway未自动部署：

1. 在Railway Dashboard中手动触发部署
2. 或者推送一个新的提交触发部署：
```bash
git commit --allow-empty -m "Trigger Railway deployment"
git push origin main
```

### 方案2: 手动创建pgboss schema

如果pgboss schema不存在，在Supabase SQL Editor中执行：

```sql
-- 创建pgboss schema
CREATE SCHEMA IF NOT EXISTS pgboss;

-- 授予权限
GRANT ALL ON SCHEMA pgboss TO postgres;
GRANT ALL ON SCHEMA pgboss TO "postgres.ynbaatdsceqtqwmqhlgu";
```

然后重启Railway服务。

### 方案3: 检查并添加缺失的环境变量

确保Railway中有以下环境变量：

**必需的变量**:
- `NODE_ENV=production`
- `PORT=3000`
- `DB_HOST=aws-0-ap-southeast-1.pooler.supabase.com`
- `DB_PORT=6543`
- `DB_NAME=postgres`
- `DB_USER=postgres.ynbaatdsceqtqwmqhlgu`
- `DB_PASSWORD=KhpGTR6dMFzZz7qq`
- `JWT_SECRET=u7uEwPvtAxpLo1GafP/WiQhTpzT+9UDOXy5nmuuQ1fU=`
- `DEMO_MODE=true` ⚠️ **重要**

**可选但推荐的变量**:
- `LLM_API_URL`
- `LLM_API_KEY`
- `LLM_MODEL`
- `JINA_READER_API_URL`
- 其他配置变量

### 方案4: 查看详细错误日志

1. 在Railway中查看实时日志
2. 尝试创建任务
3. 观察日志输出
4. 根据错误信息采取相应措施

### 方案5: 使用直连端口而非Pooler

如果Connection Pooler有问题，修改Railway环境变量：

```
DB_HOST=aws-0-ap-southeast-1.pooler.supabase.com
DB_PORT=5432  # 改为直连端口
```

注意：直连端口有连接数限制，但对于测试足够了。
