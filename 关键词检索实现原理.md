# 关键词检索实现原理

## 🎯 核心机制：LLM智能搜索

当前系统使用 **LLM（大语言模型）** 来搜索和理解关键词，而不是简单的文本匹配。

## 📊 完整执行流程

### 1. 任务触发
```
用户创建任务 → TaskScheduler.executeTask()
```

### 2. 网站内容获取（ContentRetriever）

#### 步骤2.1: 网站分析（可选）
```typescript
// 如果 ENABLE_WEBSITE_ANALYZER=true
WebsiteAnalyzer.analyze(websiteUrl)
  ↓
返回：
- pageLinks: 相关页面链接（如政策页面）
- documentLinks: 文档链接（PDF等）
```

#### 步骤2.2: 内容获取
```typescript
// 主页面
mainPageContent = fetchPageContent(websiteUrl)  // 使用Jina Reader

// 子页面（如果有）
subPageContents = fetchSubPages(pageLinks)  // 使用Jina Reader

// 文档（如果有）
documentContents = fetchDocuments(documentLinks)  // 使用Jina Reader
```

**Jina Reader的作用**:
- 支持JavaScript渲染（动态网站）
- 自动提取主要内容
- 支持PDF等文档格式
- URL格式: `https://r.jina.ai/[目标URL]`

### 3. LLM智能搜索（核心）

#### 步骤3.1: 构建提示词
```typescript
buildLLMSearchPrompt(websiteUrl, keywords, mainPageContent, subPageContents, documentContents)
```

**提示词结构**:
```
目标网站: https://www.nasdaq.com/
搜索关键词: professional

## 主页面内容:
[主页面文本内容，最多5000字符]

## 子页面内容:
### 子页面 1 (URL):
[子页面文本内容，最多3000字符]

### 子页面 2 (URL):
[子页面文本内容，最多3000字符]

## 文档内容:
### 文档 1 (URL):
[文档文本内容，最多3000字符]
```

#### 步骤3.2: 调用LLM API
```typescript
callLLM(prompt)
  ↓
发送到: Webull Claude API
模型: us.anthropic.claude-sonnet-4-20250514-v1:0
超时: 120秒
```

**系统提示词**:
```
你是一个金融合规政策分析专家。
请在[网站]域名下搜索[关键词]相关内容，
总结每个关键词的定义/解释/描述，
并以JSON格式返回总结内容。

# 技能：
1. 调用Jina Reader读取网站信息
2. 在金融语境下理解关键词
3. 在查找结果中分析关键词的定义/解释/描述

# 输出格式要求：
{
  "keywordResults": [
    {
      "keyword": "关键词1",
      "found": true,
      "content": "关键词的完整定义和解释...",
      "sourceUrl": "https://example.com/page1",
      "context": "在此页面中，关键词1被定义为..."
    }
  ]
}
```

#### 步骤3.3: 解析LLM响应
```typescript
parseLLMSearchResponse(llmResponse, keywords, subPageContents, documentContents)
  ↓
返回：
- keywordMatches: 关键词匹配结果
- documentResults: 文档检索结果
```

### 4. 保存结果到数据库
```typescript
// 为每个网站 × 每个关键词创建一条记录
retrievalResultRepository.create({
  executionId,
  websiteUrl,
  keyword,
  found: true/false,
  content: "关键词的定义和解释",
  context: "上下文片段",
  sourceUrl: "实际找到内容的URL",
  documentUrl: "如果在文档中找到"
})
```

### 5. 生成分析报告
```typescript
// 摘要文档
AnalysisService.generateSummary(executionId, retrievalResults)

// 对比报告（与上次执行对比）
AnalysisService.compareResults(executionId, taskId)

// 跨站点分析（多个网站的对比）
AnalysisService.analyzeCrossSite(executionId, retrievalResults)
```

## 🔑 关键技术点

### 1. LLM vs 简单文本匹配

**简单文本匹配**（旧方法）:
```typescript
// 只能找到精确匹配
content.includes("professional")  // 找到 "professional"
content.match(/professional/gi)   // 找到所有出现位置
```

**LLM智能搜索**（当前方法）:
```typescript
// 理解语义和上下文
LLM能够：
- 理解关键词的含义
- 找到相关的定义和解释
- 识别同义词和相关概念
- 提取完整的上下文
- 判断内容的相关性
```

**示例对比**:

关键词: "professional"

**简单匹配**:
```
找到: "professional services"
找到: "professional standards"
找到: "professional conduct"
→ 只返回包含词的句子
```

**LLM搜索**:
```
找到并理解:
"Professional refers to individuals who possess specialized 
knowledge and skills in a particular field, adhering to 
established standards of conduct and ethics..."

→ 返回完整的定义和解释
→ 理解金融合规语境
→ 提取关键信息
```

### 2. 回退机制

如果LLM调用失败，系统会自动回退到简单匹配：

```typescript
try {
  // 尝试LLM搜索
  return await llmWebSearch(...)
} catch (error) {
  console.log('LLM search failed, falling back to simple keyword matching')
  // 回退到简单文本匹配
  return fallbackKeywordSearch(...)
}
```

### 3. 错误容忍

系统设计为高容错：
- 单个网站失败不影响其他网站
- 单个文档失败不影响其他文档
- LLM失败会回退到简单匹配
- 所有错误都会被记录到数据库

```
          └─→ parseLLMSearchResponse()
              └─→ 返回: KeywordMatch[]
```

## 💡 优势和局限

### 优势
1. **智能理解**: LLM能理解关键词的语义和上下文
2. **完整提取**: 提取完整的定义和解释，不只是片段
3. **金融专业**: 系统提示词针对金融合规场景优化
4. **容错性强**: 多层回退机制，单点失败不影响整体
5. **可扩展**: 支持多网站并行处理

### 局限
1. **响应时间**: LLM调用较慢（30-120秒）
2. **成本**: 每次调用LLM API有成本
3. **依赖性**: 依赖外部LLM API的稳定性
4. **内容限制**: 发送给LLM的内容有长度限制

## 🚀 未来优化方向

### 1. 混合搜索
```
简单关键词 → 使用文本匹配（快速）
复杂概念 → 使用LLM搜索（准确）
```

### 2. 缓存机制
```
相同网站 + 相同关键词 → 使用缓存结果
避免重复的LLM调用
```

### 3. 流式处理
```
边获取内容边调用LLM
减少总体等待时间
```

### 4. 多LLM支持
```
主LLM: Claude (准确但慢)
备用LLM: GPT-3.5 (快速但可能不够准确)
根据场景自动选择
```

当前系统使用 **LLM智能搜索** 而不是简单的文本匹配，这使得系统能够：
- 理解关键词的语义
