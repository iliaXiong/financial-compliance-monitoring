# 部署问题修复总结

## 已解决的问题

### 1. CORS错误 ✅
**问题**: 前端访问后端API时被CORS策略阻止
**原因**: 后端CORS配置只允许生产域名，不支持Vercel预览部署
**解决方案**: 修改 `backend/src/index.pgboss.ts` 的CORS配置，使用正则表达式匹配所有Vercel预览域名
```javascript
origin.match(/^https:\/\/financial-compliance-monitoring.*\.vercel\.app$/)
```

### 2. 认证错误 ✅
**问题**: 前端显示"未提供认证令牌"
**原因**: 后端要求JWT认证，但前端还没有实现登录系统
**解决方案**: 在Railway中添加环境变量 `DEMO_MODE=true`，启用演示模式跳过认证

### 3. 数据库约束违反 ✅
**问题**: 创建任务时显示"数据库约束违反"
**原因**: 演示用户ID在数据库中不存在，违反外键约束
**解决方案**: 在Supabase SQL Editor中执行SQL创建演示用户
```sql
INSERT INTO users (id, email, password_hash, name, created_at, updated_at)
VALUES (
  '00000000-0000-0000-0000-000000000000',
  'demo@example.com',
  '$2b$10$dummyhashfordemopurposesonly',
  'Demo User',
  CURRENT_TIMESTAMP,
  CURRENT_TIMESTAMP
)
ON CONFLICT (id) DO NOTHING;
```

### 4. 服务器内部错误 ✅
**问题**: 创建任务时显示"服务器内部错误"
**原因**: `tasks.ts` 路由创建了新的TaskScheduler实例但没有启动，导致调用 `scheduleTask()` 时抛出错误
**解决方案**: 
- 修改 `tasks.ts` 使用依赖注入模式
- 从 `index.pgboss.ts` 注入已启动的TaskScheduler实例
- 确保整个应用使用同一个TaskScheduler实例

## 技术说明

### 演示模式
当 `DEMO_MODE=true` 时：
- 所有API请求自动使用演示用户（ID: `00000000-0000-0000-0000-000000000000`）
- 无需JWT令牌即可访问所有功能
- 适合演示和测试，不适合生产环境

### TaskScheduler单例模式
- 整个应用只有一个TaskScheduler实例
- 在 `index.pgboss.ts` 中创建并启动
- 通过依赖注入传递给路由
- 避免重复初始化和资源浪费

### CORS配置
- 支持生产域名: `https://financial-compliance-monitoring.vercel.app`
- 支持所有预览部署: `https://financial-compliance-monitoring-*.vercel.app`
- 使用正则表达式动态匹配
